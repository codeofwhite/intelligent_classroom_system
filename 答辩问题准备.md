# 智能课堂系统 - 毕业设计答辩问题准备

---

## 一、系统架构类

### Q1: 为什么选择微服务架构（3个后端服务）而不是单体应用？

**答：** 本系统采用三个独立的 Flask 服务，按职责划分：

| 服务 | 端口 | 职责 |
|------|------|------|
| model-inference-service | 5002 | 核心推理服务（视频分析、AI Agent、报告生成） |
| face-recognition-service | 5003 | 人脸签到（独立的 face_recognition 库加载） |
| user-center-service | 5001 | 用户中心（登录、权限、学生/家长/教师信息） |

选择微服务架构的原因：
1. **依赖隔离**：model-inference-service 依赖 PyTorch/ultralytics 等重型深度学习库，face-recognition-service 依赖 dlib/face_recognition，user-center-service 只需要轻量的 pymysql。如果合并，单个容器镜像会非常庞大（>5GB），拆分后各服务镜像独立，部署更灵活。
2. **进程隔离**：视频推理是 CPU/GPU 密集型操作，如果和用户登录放在一起，推理时会导致登录接口卡顿。拆分后互不影响。
3. **可独立扩展**：未来如果有多个教室，可以水平扩展 model-inference-service 实例。

服务间通信采用 HTTP REST 调用，例如 face.py 中通过 `requests.post("http://localhost:5003/reload_faces")` 通知人脸服务刷新缓存。所有服务共享同一个 MySQL 数据库和 MinIO 对象存储。

### Q2: 系统的技术栈选择理由是什么？

**答：**
- **后端选择 Flask**：相比 Django，Flask 更轻量灵活，适合快速构建 API 服务。本系统主要是 REST API，不需要 Django 的 ORM 和 Admin 等重量级功能。
- **前端选择 Vue 3 + Element Plus**：Vue 3 的 Composition API 适合构建复杂的状态管理（如实时视频流、统计数据更新），Element Plus 提供了开箱即用的 UI 组件，加速开发。
- **视觉算法选择 YOLOv8**：YOLO 系列是目前最主流的实时目标检测框架，YOLOv8 相比 YOLOv5 精度更高、推理速度更快，且 ultralytics 库的 API 设计简洁。
- **大模型选择通义千问（DashScope）**：阿里云提供的大模型 API，支持 Function Calling，国内访问延迟低，有免费额度适合学生项目。
- **对象存储选择 MinIO**：兼容 S3 协议，私有化部署，适合存储课堂视频、分析报告等大文件，不依赖第三方云存储。

### Q3: Docker Compose 部署了哪些服务？

**答：** Docker Compose 编排了以下服务：MySQL（数据库）、MinIO（对象存储）、model-inference-service（核心推理）、face-recognition-service（人脸签到）、user-center-service（用户中心），以及前端 Vue 应用的 Nginx 反向代理。各服务通过 Docker 网络互联，使用 `.env` 文件管理敏感配置（数据库密码、API Key 等）。

## 二、计算机视觉/算法类

### Q4: 为什么选择 YOLOv8 而不是其他目标检测模型？

**答：**
1. **速度与精度平衡**：YOLOv8 是单阶段检测器，推理速度快（实时视频场景下可达到 25+ FPS），满足课堂实时监测需求。而两阶段检测器（如 Faster R-CNN）精度虽高但速度慢。
2. **生态完善**：ultralytics 库提供统一的 train/predict/export API，支持导出 OpenVINO/ONNX/TensorRT 等多种推理格式，方便部署。
3. **社区活跃**：YOLOv8 有大量预训练权重和社区支持，迁移学习效果好。

本系统支持两种检测模式：
- **YOLO 检测模式**（`best_last_openvino_model`）：6 类行为检测（举手、看书、写字、使用手机、低头、睡觉），使用 OpenVINO 加速推理。
- **Pose 检测模式**（`yolov8n-pose_openvino_model`）：通过关键点估计判断姿态，识别正常坐姿、举手、低头 3 类。

### Q5: 多目标跟踪用的 ByteTrack，它的工作原理是什么？

**答：** ByteTrack 的核心创新是**不丢弃低置信度检测框**。传统跟踪器（如 DeepSORT）只用高置信度检测做关联，而 ByteTrack 将检测结果分为高分和低分两组：

1. **第一轮关联**：用高分检测框与已有轨迹做 IoU 匹配（卡尔曼滤波预测位置）。
2. **第二轮关联**：未匹配的轨迹再与低分检测框做 IoU 匹配，找回被遮挡或模糊的目标。

这样做的好处是：在课堂场景中，学生之间经常互相遮挡，ByteTrack 能更好地维持 ID 的连续性。

在代码中，`tracker.update(det[:, :5], img_size, img_size)` 接收检测框的 `[x1, y1, x2, y2, score]`，输出跟踪结果 `t.tlbr`（边界框）和 `t.track_id`（唯一 ID）。

### Q6: 行为识别的6类行为是如何定义的？"低头"和"看书"如何区分？

**答：** 6 类行为定义如下：
- **专注类（focus）**：举手（Raising-Hand）、看书（Reading）、写字（Writing）
- **分心类（distract）**：使用手机（Useing-Phone）、低头做其他事情（Head-down）、睡觉（Sleep）

"低头"和"看书"的区分依赖于**训练数据的标注标准**：
- **看书（Reading）**：学生低头但面前有书本/课本，头部角度适中（约 15-30 度俯角），属于正常学习行为。
- **低头（Head-down）**：头部大幅低垂（>45 度），且没有明显的学习工具在视野内，通常是在做与课堂无关的事情。

这个区分在 YOLO 模型的训练数据中通过标注来学习，模型会学习到头部角度、身体姿态、周围物体等综合特征。不过在实际场景中确实存在一定的混淆，这也是系统的一个局限性。

### Q7: 专注度（focus_rate）的计算公式是什么？

**答：** 专注度的计算有两种方式：

**实时模式**（realtime.py）：
```python
focus_rate = 100 * (hand_up + study_norm) / total
# 其中 total = hand_up + study_norm + look_down + abnormal
```

**离线报告模式**（reports.py 的 focus_trend）：
```
将视频总帧数分成 N 个时间窗口（默认10段）
每段内：focus_rate = 专注行为帧数 / 总检测帧数 × 100%
```

专注行为包括：举手、看书、写字；分心行为包括：使用手机、低头、睡觉。

### Q8: 关键帧选取策略是什么？

**答：** 系统采用**双策略关键帧选取**：

1. **分心阈值触发**：当分心人数达到 `GLOBAL_DISTRACT_NUM`（默认2人）且距离上次拍摄超过 `KEY_FRAME_INTERVAL`（默认30帧）时，保存当前帧为关键帧。这能捕捉到课堂中最混乱的瞬间。
2. **定期采样**：每隔 `KEY_FRAME_INTERVAL * 3`（至少90帧）自动拍摄一张关键帧，不管是否有分心行为。这保证了正常课堂也有关键帧记录。

最终在报告生成时，**优先选取分心人数最多的那张**作为代表性关键帧上传到 MinIO。如果没有关键帧（极端情况），从视频中间截取一帧作为兜底。

### Q9: 模型的推理加速方案是什么？

**答：** 本系统使用 **OpenVINO** 进行模型推理加速。OpenVINO 是英特尔推出的推理优化工具，支持将 PyTorch/YOLOv8 模型转换为 IR（Intermediate Representation）格式，利用 CPU 的 SIMD 指令集（AVX2/AVX-512）加速推理。

在代码中，模型加载时指定 `task` 参数：
```python
shared.model = YOLO(model_path, task=cfg['task'])
```
推理时使用 `half=True` 启用半精度浮点计算（FP16），进一步提升速度。

## 三、人脸识别/签到类

### Q10: 人脸签到用的是什么算法？

**答：** 人脸签到使用的是 `face_recognition` 库，其底层基于 **dlib** 的人脸识别算法：
1. **人脸检测**：使用 HOG（Histogram of Oriented Gradients）+ SVM 分类器或 CNN 模型定位人脸位置。
2. **特征提取**：使用 dlib 的 ResNet 模型（基于 128 维特征向量）将人脸编码为 128 维向量。
3. **匹配识别**：计算待识别人脸与已知人脸库的欧氏距离，距离最小且小于阈值（0.5）则识别成功。

在 pose 模式下，代码中的人脸识别逻辑如下：
```python
dists = face_recognition.face_distance(pose_utils.known_encodings, enc)
if len(dists) > 0 and dists.min() < 0.5:
    sid = pose_utils.known_ids[np.argmin(dists)]
```

### Q11: 签到记录的存储方式是什么？为什么没有用数据库？

**答：** 当前签到记录存储在 face-recognition-service 的本地文本文件 `sign_log.txt` 中，每行格式为 `学生学号 时间`。每次开始新签到时会清空旧记录。

**这是当前系统的一个不足之处**。选择文本文件的初衷是：
1. 签到功能运行在独立的 face-recognition-service 中，该服务设计为轻量级，不直接连接数据库。
2. 签到是一次性、短时间的行为（一堂课开始时启动，结束后获取结果），文本文件足够应对。

**改进方向**：可以将签到记录持久化到 MySQL 数据库中，设计签到记录表包含 `student_code`、`class_code`、`sign_time`、`lesson_section` 等字段，支持按日期查询历史签到记录。

### Q12: 人脸数据库更新后如何同步？

**答：** 系统支持通过 `POST /api/face/refresh_cache` 接口一键刷新人脸缓存。该接口会：
1. 调用 face-recognition-service 的 `/reload_faces` 接口，从 MinIO 重新下载所有人脸图片并提取编码。
2. 调用 model-inference-service 本地的 `_sync_all_from_minio()`，从 MinIO 同步到本地 `face_lib` 目录供 pose 模型使用。

人脸照片统一存储在 MinIO 的 `face-images` bucket 中，路径格式为 `{class_code}/{student_code}/{filename}`，数据库 `face_images` 表记录了每张照片的元数据。

## 四、数据库设计类

### Q13: 数据库总共多少张表？表之间的关系是怎样的？

**答：** 共 **13 张核心表**，按功能域分为 5 组：

**用户管理域（5张）**：`users`（用户基础表）、`classes`（班级表）、`teachers`（教师扩展）、`students`（学生扩展）、`parents`（家长扩展）。通过 `user_code` 关联，实现一个用户表 + 多个角色扩展表的设计，避免表冗余。

**课堂报告域（2张）**：`course_reports`（课堂报告主表，存储 MinIO 路径）、`student_reports`（学生个人行为报告，存储行为计数和 AI 评语）。

**人脸签到域（2张）**：`face_images`（人脸照片元数据）、`face_student_mapping`（人脸 ID 与学生映射）。

**课程排期域（1张）**：`teacher_course_schedule`（教师课程表）。

**AI 智能域（3张）**：`chat_sessions`（AI 对话会话，JSON 存储消息历史）、`agent_logs`（Agent 调用日志）、`teacher_long_memory`（教师长期记忆）。

### Q14: course_reports 和 student_reports 之间是什么关系？

**答：** `course_reports` 是课堂维度的报告（一节课一条记录），`student_reports` 是学生维度的报告（一个学生在一节课中的表现）。理论上是一对多关系，但当前设计中 `student_reports` 的 `report_code` 字段是可选的，没有建立强制外键关联。

这是因为两种报告的生成流程不同：
- `course_reports` 在视频分析停止时自动创建，存储整体统计。
- `student_reports` 由教师在报告详情页手动创建/编辑，可以添加教师评分和评语。

通过 `class_code` + `lesson_time` 可以间接关联两者。

### Q15: 为什么用 MinIO 存储视频和报告，而不是直接存数据库？

**答：** 
1. **文件大小**：课堂视频通常几百 MB 甚至几 GB，不适合存入 MySQL 的 BLOB 字段，会严重影响数据库性能。
2. **访问模式**：视频和大文件适合通过 HTTP 流式访问，MinIO 支持预签名 URL，前端可以直接播放视频而不需要经过后端转发。
3. **成本**：MinIO 是开源的对象存储，兼容 S3 协议，可私有化部署，不产生云存储费用。

MySQL 只存储文件的路径（`minio_video_path` 等字段），通过路径去 MinIO 获取实际文件。删除报告时会同时清理 MinIO 中的文件和数据库记录。

### Q16: chat_sessions 用 JSON 存对话历史，会不会有性能问题？

**答：** 当前设计中，`chat_sessions.messages` 字段使用 MySQL 的 JSON 类型存储完整的对话历史数组。对于教师端的 AI 助手场景，每个会话的消息量通常在几十条以内（最大循环8轮），JSON 数据量不会太大（几 KB 到几十 KB），性能可以接受。

但如果未来消息量增大，可以考虑以下优化：
1. 将历史消息迁移到独立的 `chat_messages` 表，每条消息一行。
2. 对旧消息进行归档压缩，只保留最近 N 条在 JSON 中。
3. 使用 Redis 缓存活跃会话的消息。

## 五、AI/大模型类

### Q17: AI Agent 的 Function Calling 是怎么实现的？

**答：** 系统实现了一个基于通义千问大模型的 **Function Calling Agent**，核心流程如下：

1. **意图识别**：先用 `qwen-turbo` 对用户问题做意图分类（query_history / query_report / query_class / chat_chitchat / other），闲聊和无关问题直接拦截返回。
2. **工具定义**：定义了 11 个工具函数（TOOLS），包括查询历史报告、获取单节课详情、获取班级学生列表、获取专注度排行榜、查询学生跨课堂行为汇总等。
3. **多轮推理**：将问题和工具定义发送给 `qwen-plus` 模型，模型决定调用哪个工具及参数。系统执行工具后将结果返回给模型，模型判断是否需要继续调用工具或生成最终回答。
4. **最大循环**：最多 8 轮工具调用，防止无限循环。
5. **日志记录**：每次调用都记录到 `agent_logs` 表，包含意图、工具调用链、参数、结果和最终回答。

```python
while current_round < max_round:
    resp = dashscope.Generation.call(
        model="qwen-plus",
        messages=messages,
        tools=TOOLS,
        result_format="message"
    )
    if "tool_calls" in ai_msg:
        # 执行工具，追加结果到 messages
        ...
    else:
        # 获取最终答案
        answer = ai_msg.get("content")
        break
```

### Q18: 教师长期记忆机制是怎么工作的？

**答：** `teacher_long_memory` 表为每位教师维护一个使用画像，包含：
- **常关注班级**（focus_class_codes）：教师经常查询的班级，去重保留最近10个。
- **常查看报告**（focus_report_codes）：教师经常查看的报告。
- **常查询学生**（focus_student_codes）：教师经常查询的学生。
- **最近使用班级**（last_class_code）：上次对话涉及的班级。
- **关注话题偏好**（prefer_focus_topic）：根据工具调用自动推断（如常查排行榜则标记为"专注度排行"）。
- **最近问题**（recent_queries）：滚动窗口保留最近5条问题。
- **累计查询次数**（query_count）：每次工具调用 +1。

这些信息会被注入到 System Prompt 中：
```
【用户长期习惯画像】
教师工号：T2025001
累计互动：15 次
常关注班级：C001,C002
最近使用班级：C001
关注话题：专注度排行
最近提问：「C001班谁最专注」「张三最近表现怎么样」
请结合以上习惯信息，优先使用该老师常用的班级和报告数据回答。
```

这样 AI 就能理解"我们班""上次那个学生"等指代性问题。

### Q19: AI 评语生成和家校共育建议是怎么做的？

**答：** 

**AI 评语**（`/api/ai/analyze`）：将学生的课堂行为数据（正常坐姿次数、举手次数、低头次数、专注度百分比）构造 Prompt，调用 `qwen-turbo` 生成 80 字左右的温和鼓励性评语，适合家长阅读。

**家校共育建议**（`/api/ai/advice`）：查询该学生所有历史报告，统计平均专注度、累计坐姿达标次数、举手次数、低头次数，生成结构化的总结和建议，包含 4 条具体的家庭配合建议。

**课堂 AI 分析报告**（`/api/generate_and_save_ai`）：调用 `ai_agent.analyze_class_report`，综合行为统计、课堂信息和关键帧图片（多模态），生成完整的课堂分析报告，以 Markdown 格式保存到 MinIO。

## 六、工程实践/性能类

### Q20: 实时视频分析的帧率（FPS）是多少？

**答：** 实时分析的帧率取决于多个因素：
- **模型推理**：YOLOv8 + OpenVINO 在 CPU 上推理约 20-30ms/帧（640×640 输入），即约 30-50 FPS。
- **ByteTrack 跟踪**：卡尔曼滤波 + 匈牙利算法匹配，开销很小（<5ms）。
- **视频 I/O**：MJPEG 编码和网络传输。

综合来看，单路视频流的实时分析帧率约为 **15-25 FPS**，基本满足课堂实时监测需求。如果需要更高帧率，可以：
1. 降低输入分辨率（如从 640 降到 320）。
2. 使用 GPU 推理（需要 NVIDIA 显卡 + CUDA）。
3. 跳帧检测（每隔 N 帧检测一次，中间帧复用跟踪结果）。

### Q21: 录制过程中的状态管理是怎么做的？有什么局限性？

**答：** 录制状态使用模块级全局字典 `_state` 管理，包含：
- `is_recording`：是否正在录制
- `behavior_data`：行为轨迹数据列表
- `student_behaviors`：每个 track_id 的行为统计
- `total_count`：全局行为计数
- `keyframe_records`：关键帧记录
- `real_time_stats`：实时统计数据

**局限性**：
1. **单实例**：全局字典在内存中，只支持一路视频同时录制。如果需要多教室同时录制，需要改造为独立的录制任务实例。
2. **不持久化**：如果服务重启，未完成录制的数据会丢失。
3. **内存占用**：长时间录制的行为数据全部存在内存中，如果一节课 45 分钟、每帧多个检测目标，`behavior_data` 可能积累数万条记录。

**改进方向**：可以使用 Redis 或消息队列缓存行为数据，定期批量写入数据库；或者使用 Celery 异步任务管理录制会话。

### Q22: ByteTrack 的 track_history 用 deque(maxlen=15) 做行为平滑，为什么选15？

**答：** `deque(maxlen=15)` 用于**行为标签的时间平滑**。每个跟踪目标维护最近 15 帧的行为标签，通过投票（`max(set(...), key=...)`）确定最终标签。

选择 15 帧的原因：
- 视频帧率为 25 FPS，15 帧约 0.6 秒的时间窗口。
- 这个窗口足够短，能快速响应行为变化（如从看书变为举手）。
- 又足够长，能过滤掉偶发的误检测（单帧误检不会影响最终结果）。

如果窗口太小（如 3 帧），容易受噪声影响；太大（如 30 帧），行为切换会有明显延迟。15 是一个经验性的平衡值。

### Q23: 专注度趋势是怎么计算的？

**答：** `/api/report/focus_trend` 接口根据行为轨迹 CSV 计算专注度趋势：

1. 从 MinIO 下载该节课的行为轨迹 CSV（`tracks.csv`），每行包含 `frame_id`、`behavior_label`。
2. 将总帧数分成 N 个时间窗口（默认 10 段，可配置）。
3. 每个窗口内统计专注行为帧数占比，得到该时间点的专注度百分比。
4. 返回 `{labels: ["0%", "10%", ...], data: [85, 82, 78, ...]}`，前端用折线图展示。

这样教师可以直观看到一节课中专注度随时间的变化趋势，比如后半节课专注度下降。

## 七、前端/用户体验类

### Q24: 三个前端入口分别是做什么的？

**答：**

| 前端 | 目标用户 | 主要功能 |
|------|----------|----------|
| **admin**（教师管理端） | 教师 | 查看课堂报告、管理学生人脸、AI 对话助手、课程表管理、学生行为分析 |
| **client**（学生/家长端） | 学生和家长 | 查看个人课堂表现、历史报告、AI 评语、学习助手对话 |
| **terminal**（教室采集端） | 教室设备 | 摄像头推流、实时签到、录制控制 |

数据流：
- **terminal** 摄像头采集视频 → 推流到 model-inference-service 进行实时分析
- **admin** 调用 start_record/stop_record 控制录制 → 查看分析报告
- **client** 查询 student_reports 查看个人表现

### Q25: 视频流用的什么方案？

**答：** 使用 **MJPEG（Motion JPEG）** 方案，通过 HTTP 的 `multipart/x-mixed-replace` 协议实现：

```python
@app.route('/video_feed')
def video_feed():
    return Response(generate_frames(),
                    mimetype='multipart/x-mixed-replace; boundary=frame')
```

`generate_frames()` 是一个生成器，每处理一帧就 yield 一个 JPEG 图片。前端直接用 `<img src="/video_feed">` 就能显示实时视频流。

**为什么不用 WebSocket 或 WebRTC？**
- MJPEG 方案实现最简单，不需要额外的信令服务器。
- 课堂场景下视频流只需要单向传输（服务器→浏览器），MJPEG 足够。
- 延迟约 0.5-1 秒，对于课堂监测场景可以接受。

**局限性**：MJPEG 没有压缩效率高的编码（每帧都是独立 JPEG），带宽占用较大。如果需要更低延迟和更高效的传输，可以考虑 WebRTC。

## 八、创新点与不足

### Q26: 你这个系统的创新点是什么？

**答：** 
1. **多模态课堂分析**：将 YOLO 行为检测 + ByteTrack 多目标跟踪 + 人脸识别 + 大模型 Agent 融合，实现了从"检测→跟踪→识别→分析→报告→对话查询"的完整闭环。
2. **AI Agent 对话式数据查询**：教师可以通过自然语言对话查询课堂数据（如"C001班谁最专注"），系统通过 Function Calling 自动调用数据库查询工具，降低了数据获取门槛。
3. **教师长期记忆机制**：AI 助手能记住教师的使用习惯（常关注的班级、学生、话题），实现个性化服务。
4. **关键帧智能选取**：双策略（分心阈值触发 + 定期采样）自动选取课堂关键时刻，辅助教师快速回顾课堂亮点和问题。
5. **家校互通设计**：学生/家长端可以查看 AI 评语和家校共育建议，促进家校协同。

### Q27: 系统目前最大的不足是什么？

**答：**
1. **签到记录不持久化**：签到记录存储在文本文件中，没有存入数据库，无法支持历史签到查询。
2. **单实例限制**：实时录制使用全局字典管理状态，只能支持一路视频同时录制，不支持多教室并行。
3. **行为识别精度**：在学生密集、遮挡严重的场景下，"低头"和"看书"容易混淆，模型精度有待提升。
4. **缺少自动化测试**：虽然有 tests/ 目录，但测试覆盖率有限，缺乏端到端的集成测试。
5. **实时性能瓶颈**：纯 CPU 推理在高分辨率视频下帧率可能不足，GPU 推理的支持需要进一步完善。

### Q28: 如果再给你一个月，你会优先改进什么？

**答：** 按优先级排序：
1. **签到记录数据库化**：设计签到记录表，支持按日期、班级查询历史签到。
2. **多教室支持**：将录制状态从全局字典改为基于任务 ID 的独立实例，支持多路视频同时分析。
3. **增加自动化测试**：编写 API 集成测试和前端 E2E 测试，提高代码质量。
4. **GPU 推理优化**：支持 CUDA 加速，提升实时分析帧率。
5. **前端体验优化**：增加 WebSocket 实时推送统计数据，减少轮询。

### Q29: 有没有考虑过隐私问题？

**答：** 这是本系统需要特别关注的问题。当前系统的设计考虑了以下隐私保护措施：
1. **数据私有化**：所有数据（视频、人脸图片、分析报告）存储在私有部署的 MinIO 中，不上传到第三方云服务。
2. **内部使用**：系统仅面向学校内部使用，数据访问需要登录认证。
3. **人脸数据管理**：人脸照片需要教师主动上传和管理，不进行大规模人脸采集。

**不足之处**：当前系统缺乏细粒度的权限控制（如学生只能查看自己的数据、家长只能查看自己孩子的数据），也缺乏数据生命周期管理（如自动删除过期视频）。在正式部署前，需要与学校协商制定数据使用规范和隐私保护方案。

## 九、技术细节追问

### Q30: 检测框与跟踪 ID 的匹配是怎么做的？

**答：** 在一帧中，YOLO 模型输出多个检测框 `[x1, y1, x2, y2, score, class_id]`，ByteTrack 输出多个跟踪目标 `[x1, y1, x2, y2, track_id]`。由于两者的数量和位置可能不完全一致，需要将每个跟踪目标与最近的检测框匹配，以获取该目标的行为类别：

```python
best_cls = 0
min_dist = 1e9
for d in det:
    dcx = (d[0]+d[2])/2
    dcy = (d[1]+d[2])/2
    dist = (dcx - cx)**2 + (dcy - cy)**2
    if dist < min_dist:
        min_dist = dist
        best_cls = int(d[5])
```

使用跟踪框中心点与检测框中心点的欧氏距离，选取最近的检测框的类别作为该跟踪目标的行为标签。

### Q31: 模型切换是怎么实现的？

**答：** `POST /switch_model` 接口接收模型名称，从 `MODEL_CONFIGS` 配置字典中获取对应的标签定义和任务类型，然后重新加载 YOLO 模型：

```python
cfg = MODEL_CONFIGS[target_model]
shared.model = YOLO(model_path, task=cfg['task'])
shared.current_model_config = cfg
```

切换后，后续的视频分析会使用新模型的标签和检测逻辑。系统支持在运行时动态切换，不需要重启服务。

### Q32: 为什么同时使用 dashscope SDK 和 openai SDK 调用大模型？

**答：** 系统中两种调用方式并存：
- **openai SDK**（ai_bp.py）：通过 OpenAI 兼容接口调用 `qwen-turbo`，使用 `base_url="https://dashscope.aliyuncs.com/compatible-mode/v1"`。
- **dashscope SDK**（chat_agent.py、chat_bp.py）：直接使用阿里云的 DashScope SDK 调用 `qwen-turbo` 和 `qwen-plus`。

两种方式本质上调用的是同一个底层模型，区别在于：
- dashscope SDK 原生支持 Function Calling（`tools` 参数），chat_agent 依赖此能力。
- openai SDK 的兼容模式更简洁，适合简单的文本生成场景。

这确实是代码风格不统一的问题，理想情况下应该统一使用一种调用方式。
